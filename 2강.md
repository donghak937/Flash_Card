## 2.1

- What is UML?
    - 여러 객체지향 표기법의 통합판
- Functional model
    - **UML Notation: Use case diagrams**
    
    ![image.png](image.png)
    
- Dynamic model
    - UML Notation: Sequence, State chart and Activity diagrams

- Use case vs Class
    - **Use case**: 기능 관점, 사용자 입장
    - **Class**: 구조 관점, 객체/속성/관계
- Sequence vs Statechart
    - **Sequence**: 객체들끼리 메시지를 주고받는 순서
    - **Statechart**: 객체 하나가 상태를 어떻게 바꾸는지
- Sequence vs Activity
    - **Sequence**: 객체 간 상호작용 중심
    - **Activity**: 작업 절차/흐름 중심
- Class diagram
    - 내부구조 설명
    - 어떤 클래스, 객체가 있는지
- Interaction Diagrams
    - 객체 사이에 일어나는 상호작용
    - 누가 먼저 요청하는지, 누가 응답하는지, 어떤 순서인지
- State Machine Diagrams
    - 개별 객체 하나의 동적 행동
    - 객체 하나가 어떤 상태에 있고, 이벤트 때문에 어디로 이동하나?
- **An activity diagram**
    - **activities 의 관점에서 행동 묘사**
- **activities**
    - 연산들의 실행
- FlowChart랑 비슷함

## 2.3

- System
    - 전체임
    - 자연 시스템
    - 인공 시스템 (비행기)
- Sub Systems
    - 더 단순한거
    - 엔진 시스템
    - 브레이크 시스템
- Modeling
    - 시스템 복잡하니 모델로 보자(추상화 만들기)
- divide-and-conquer
    - 복잡 시스템을 쪼개기
    - 사람이 이해 가능한 정도로 쪼개기
    - 많아도 5 ~ 9
- 만들어진 모델의 집합 → the system model
- view
    - 모델의 일부
- Data Type
    - 같은 type의 집합
    - 어떤 연산을 할 수 있는지도 정의
- An abstract data type
    - 어떻게가 아니라 뭘 제공하는지
    - 집합이 어레이든 링크드리스트든 일단 집합을 사용한다.
- abstract class
    - 공통점 모아두는 부모 클래스
    - 하지만 그 자체로는 객체를 만들지 않는 클래스
    - 이탈릭으로 쓰기
    - 복잡성 줄이기 + reuse
- Event Class
    - 사건의 종류
    - ButtonPressed() → event class
    - 사용자가 눌렀다. → event
        - Event와 message는 인스턴스
        - 메세지는 다른 객체의 메서드 실행 요청
- application domain
    - 문제세계
    - 실제 비행기
- solution domain
    - 해결하기 위한 세계
    - 교통제어 시스템
- **Falsification**
    - 모델이 틀렸다는 것을 드러내기
- Prototyping
    - 사용자에게 평가 → **Falsification**

    - 유일한 이름
    - Actors
    - **이 기능을 시작하려면 먼저 뭐가 되어 있어야 하냐**
    - 사건 흐름 (번호 매기기)
    - 두 개 열
    - 종료조건
    - Quality requirements
        - 응답 시간 2초 이내
        - 보안 필요
    - 자연어로
    
    ![image.png](image%201.png)
    
- Communication
    - Actor ↔ Use Case 사이에 정보 오갈때
    - 실선으로
- Include relationships
    - 공동기능 빼서 쓰기 (Use Case 공통점 줄이기)
    - `OpenIncident` **include** `ViewMap`
    - `AllocateResources` **include** `ViewMap`
    - 점선 화살표 + <<include>>
- Extend
    - 특정 조건에서 추가 동작이 붙음
    - (예외적 행동)
    - 점선 화살표 + <<Extend>>
- Inheritance relationships
    - 상위 Use Case 추상적
    - 하위 Use Case 구체적
    - `Authenticate`
    - `AuthenticateWithPassword`
    - `AuthenticateWithCard`
    - AuthenticateWithPassword → Authenticate
- Scenarios
    - use case는 해당 기능과 관련된 가능한 모든 시나리오를 설명하는 추상화다
    - (실제 시뮬)
- include
    - c1이 c2 include
    - c1에 c2가 포함됨
- extend
    - c2가 c1을 extend
    - c1 혼자 독립 가능
    - c2는 특별한 상황에서만
- Class Diagram
    - **속성(attributes)** 과 **행동(behavior)** 을 정의한 추상화
- Association
    - 클래스끼리의 관계
- link
    - 객체 끼리의 연결
- Navigation arrow

    - 방향성이 있음
    - 화살표가 없으면 양뱡향
