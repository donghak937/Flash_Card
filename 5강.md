## An Overview of Analysis

- functional model : 무슨 기능을 제공 : use case / scenario
- analysis object model : 시스템 안에 어떤 객체 : class diagram
- Dynamic model : 시간의 흐름에 따라 시스템 : sequence diagram

## Analysis Object Models and Dynamic Models

- analysis model은 사용자 관점이다
- 사용자에게 의미 있는 개념

- analysis object model

- dynamic model

## Entity, Boundary, and Control Objects

- Entity
	- 지속적으로 추적/저장하는 정보
	- 무엇을 저장하나
- Boundary
	- 액터와 시스템 사이의 상호작용 (UI)
	- 어떻게 사용자와 만나는가
- Control
	- 유스케이스를 실현하는 흐름 제어 객체
	- 그 use case를 어떻게 진행시키나

- stereotype mechanism
	- <\<boundary>>
- naming conventions
	- control
		- control
	- boundary
		- UI
		- Button

## Identifying Entity Objects

- Natural language analysis
	- 분석가가 문장을 어떻게 쓰느냐
	- 명사는 엄청 많지만, 진짜 클래스는 그보다 훨씬 적다
- Proper noun → Instance
- Common noun → Class
- Doing verb → Operation
- Being verb → Inheritance
- Having verb → Aggregation
- Modal verb → Constraints
- Adjective → Attribute

- 추가 heuristic
	- 유스케이스를 이해하려면 반드시 명확히 해야 하는 용어
	- 유스케이스에서 반복적으로 등장하는 명사
	- 시스템이 추적해야 하는 실세계 개체
	- 시스템이 추적해야 하는 실세계 활동
	- Data sources or sinks

## Identifying Boundary Objects

- 액터(actor)와 시스템이 만나는 접점
- 사용자 인터페이스를 대충 큰 단위로 표현하는 것

- visual detail을 boundary object로 자세히 설명하면 안됨
	- 그런 건 스케치/목업이 더 적합
	- UI는 나중에도 계속 바뀐다

- use case를 시작하는 UI control 식별
- 사용자가 데이터를 입력하는 form 식별
- 시스템이 사용자에게 응답하는 notice / message 식별
- 여러 액터가 있으면 actor terminal 식별
- 시각적 aspect는 boundary로 모델링하지 말 것
- 항상 최종 사용자의 용어를 사용할 것

## Identifying Control Objects

- boundary 객체와 entity 객체 사이를 조정하는 객체
- use case를 처리하기 위해 잠깐 살아있는 객체
- Boundary에서 정보 모으고 Entity로


## Mapping Use Cases to Objects with Sequence Diagrams

- 객체 간 상호작용의 순서를 모델링
	- 새로 필요한 객체 찾기
	- 빠진 행동 찾기
	- 유스케이스 설명 보완하기
- 규칙
	- 유스케이스 시작한 액터
	- 액터가 유스케이스 시작할 때 쓰는 boundary
	- 나머지 흐름을 관리하는 control
	- 즉 시작용 boundary가 control 생성
	- boundary는 control이 만든다
	- entity는 control과 boundary가 접근 가능
	- entity는 boundary/control을 절대 접근하지 않는다\

## Identifying Associations

- 두 클래스가 서로 어떤 관계를 가지는가
- 객체들 사이 관계를 명확하게 보여줌
- boundary case를 발견하게 해줌
- 구성 요소
	- name
	- role at each end
	- multiplicity at each end
- verb, verb phrase를 보고 찾는다
- association 이름과 role 이름은 정확히 써라
- qualifier를 써서 key를 드러내라
- 다른 association으로부터 유도 가능한 건 제거해라
- multiplicity는 association 집합이 안정된 뒤에 정해라
- association이 너무 많으면 모델이 읽기 어려워진다
- identifying characteristic : 식별자 범위(namespace)

## Modeling State-Dependent Behavior of Individual Objects

- 현재 상태(state) 에 따라 할 수 있는 행동이 달라짐
- sequence diagram
	- 여러 객체에 걸쳐 behavior를 분배해서 보여줌
	- 시나리오 전체의 흐름
- state machine diagram
	- 한 개의 객체 관점에서 behavior를 표현함
	- 이 객체의 인생 사이클
- Why State machine?
	- 객체 행동을 더 형식적
	- 누락된 use case

## Reviewing the Analysis Model

- 분석 모델은 점진적이고 반복적으로 개발된다
	- 개발자끼리 먼저 검토
	- 개발자 + 고객이 같이 검토
- review의 목표 : requirements specification이 다음 조건을 만족하는지 확인
	- correct = 정확한가
	- complete = 완전한가
	- consistent = 일관성 있는가
	- unambiguous = 모호하지 않은가
	- realistic = 현실적인가
	- verifiable = 검증 가능한가
