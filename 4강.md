## Functional Requirements

- 기능 요구사항 = 시스템의 기능 목록
- 사용자가 어떤 입력을 하면 시스템이 어떤 처리를 하고 어떤 결과를 보여주는가

## Nonfunctional Requirements

- 어떤 품질을 가져야 하는가 + 어떤 제약 아래 동작해야 하는가
- 품질, 제약, 성능, 유지보수성

- Functionality

	- Dependability : 믿고 맡길 수 있는 정도
	- Robustness : 잘못된 입력, 스트레스 환경 에서도 올바르게 기능할 수 있는 정도

- Performance

	- Accuracy

	- Internationalization

## Completeness, Consistency, Clarity, Correctness

- Completeness : 빠진 게 없어야 한다
- Consistency : 앞에서 한 말과 뒤에서 한 말이 충돌하면 안 됨
- Clarity / Unambiguity : 두 가지 이상으로 해석되면 안 된다
- Correctness : 고객이 원하는 것과 맞아야 하고 개발자가 만들려고 하는 시스템과도 맞아야 함

## Realism, Verifiability, Traceability

- Realism : 말은 멋있는데 현실적으로 못 만들면 안 된다
- Verifiability : 테스트로 증명 가능해야 한다
- Traceability : 앞으로도 따라갈 수 있고, 뒤로도 거슬러 올라갈 수 있어야 함

## Identifying Actors

- Actor : 액터는 시스템과 상호작용하는 외부 개체
	- 사람(human)
	- 외부 시스템(external system)

- 시스템은 어떤 작업을 수행하는 사용자 그룹을 지원하는가?
- 어떤 사용자 그룹이 시스템의 주요 기능(main functions) 을 실행하는가?
- 어떤 사용자 그룹이 **보조 기능**(유지보수, 관리 등)을 수행하는가?
- 시스템이 어떤 외부 하드웨어/소프트웨어 시스템 과 상호작용하는가?

## Identifying Scenarios

- ### 시나리오를 찾기 위한 질문

1. 액터가 시스템에게 수행하길 원하는 태스크는 무엇인가?
2. 액터는 어떤 정보 를 접근하는가?
    - 누가 그 데이터를 생성하는가?
    - 수정 가능한가?
    - 삭제 가능한가?
    - 누가 그렇게 하는가?
3. 액터가 시스템에 알려줘야 할 외부 변화는 무엇인가?
    - 얼마나 자주?
    - 언제?
4. 시스템은 액터에게 어떤 이벤트를 알려줘야 하는가?
    - 얼마나 빠르게 알려야 하는가?

## Refining Use Cases

- 시스템이 다루는 구체적 요소
- 액터와 시스템의 더 자세한 상호작용 순서
- 접근 권한
- 빠진 예외 처리
- 여러 유스케이스에 공통인 기능 정리
- 빠진 예외 찾기
- 공통 기능 분리
- 저수준 상호작용 구체화
- 권한 명시
