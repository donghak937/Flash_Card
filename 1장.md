    - 개발 과정 관리 어려움

## 1.2

- 소프트웨어 공학은 모델링
    - 세부사항 집중
    - 일단 나머지 무시
- 문제 해결
    - 자원 한정
- 지식 습득
    - 요구사항
    - 문제영역
    - 제약조건
    - 단계순은 아니다 나중에 들어온게 처음에 들어온 지식을 밀어낼 수 있음
- 근거 중심
    - 왜 그 기술을 썼는지, 남긴다.
- 모델링
    - 복잡한 문제 해결을 위한 축소 설명판
    - 모델은 너무 크거나, 작거나 복잡하거나, 비싸면 안된다.
- 문제 해결
    - 문제 정식화 → 분석 → solutoin → decide appropriate way → Specify the solution.
- OOP 개발
    - 요구사항 도출 → 분석 → 시스템 설계 → 객체 설계 → 구현 → 테스트
- technique
    - 어떻게 할 것인가?
- Methodologies
    - 여러 방법을 묶은 개발 철학/체계
- Tool
    - IDE, 소프트웨어 도구
- Analysis
    - 문제 쪼개기
- Synthesis
    - 다시 합치기
- 폭포 모델
    - 요구사항 → 설계 → 구현 → 테스트 → 유지보수
- Risk-based development
    - 위험한거 부터
- Issue-based development
    - 순서대로 하는게 아니다 → 관리가 어려움!
- 근거 기록 중요
    - rationale = 설계 결정의 이유와 배경

    - 그들이 시스템으로 무엇을 하는지(use case) → 기능 중심 상호작용 시나리오
- Analysis

    - complete
    - consistent
    - dynamic model
        - 시스템 어떻게 움직이는지
    - object model

    - define and decompose
    - select strategies

    - 파일 저장으로 할지

    - integration testing → 합쳐서 잘 돌아 가는지.
    - system testing → 전체로 잘 되는지
